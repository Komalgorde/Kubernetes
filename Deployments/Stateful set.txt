**StatefulSet**
-Used for stateful applications (apps that need data persistence).
??Examples:
-Databases (DB)
-MongoDB
-Cassandra
??Key Features:
-Provides stable and persistent storage.
-Each pod gets a unique identity (name).
-Pod identity remains the same even after rescheduling.
?? Advantages (General)
-Scalability
-Automation
-Self-healing
-Auto scaling & updates
-Load balancing

?? Deployment Strategy (Rolling Update)
->To check explanation:
kubectl explain deployment.spec.strategy
Add in YAML:
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 100%
    maxUnavailable: 0
    
Rolling Update (Live Deployment Update)
?? Initial Concept
Multiple pods running: Version V1 need to gradualy update to v2

Goal:
#Perform live update without downtime

?? Apply Deployment
kubectl apply -f <file>.yaml

->To Watch pods in real-time:
watch kubectl get pods
?? Update Deployment (V1 ? V2)

Open another terminal / login again (e.g., EC2)
Switch user:
sudo su -
->Edit deployment:
kubectl edit deployment testpod
Change image: v1 ? v2

#In the first terminal, you will see pods updating gradually

?? Important Concept
Old version (V1) should not be deleted immediately
First: Deploy V2
Then:
Gradually remove V1

#Prevents application crash / downtime

?? YAML Configuration for Safe Update
Go to YAML file and add:
spec:
  minReadySeconds: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
#Meaning
-minReadySeconds: 10
-Pod must be ready for 10 seconds before considered available
-maxSurge: 1
-maxUnavailable: 0
-No pod goes down ? Zero downtime

?? Apply Changes Again
kubectl apply -f <file>.yaml
Check pods:
kubectl get pods
?? Rollback Scenario (V2 ? V1)
Edit again:
kubectl edit deployment testpod
Change:
v2 ? v1
-> Kubernetes will again perform rolling update
?? Behavior Observation
Every ~10 seconds:
One pod is created (new version)
One pod is terminated (old version)
